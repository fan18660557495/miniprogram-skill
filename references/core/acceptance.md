# 最小验收与评审清单

这个文件把前面的规范收敛成一组更容易执行和复查的验收项。

它不替代各专题文档，而是用于：

- 新页面或新组件落地后的最小验收
- 小程序前端代码评审时的快速检查
- skill 自身回归时的结构化自检

## 使用方式

默认按这 3 步走：

1. 先读对应专题文档，完成设计或实现。
2. 再用本文件做最小验收。
3. 如果发现问题，再回到对应专题文档修正。

## 最小通过条件

一个页面或组件至少满足这些条件，才算达到默认基线：

- 用户第一眼能理解当前页面的主任务
- 只有一个主视觉操作重心，或主次关系足够清楚
- `loading`、`empty`、`error` 至少覆盖其中与当前场景相关的状态
- 用户可见文案不暴露技术异常、接口路径、状态码或堆栈
- 提交、录音、重试等关键动作能防重复触发
- 键盘、安全区、弱网和页面切换不会直接破坏核心流程

## 默认验收阈值

下面这些阈值是默认基线，不是所有项目都必须逐像素一致。
如果项目已有明确规范，优先沿用项目规范。

- 可点击区域建议不小于 `44px`
- 常规输入区和主按钮默认高度优先使用 `48px`
- 吸底栏和底部按钮必须覆盖安全区
- 错误提示应紧贴字段或问题区域，不应漂到远处
- 关键动作在对应场景下应保持可见或可快速回到可见状态

## 页面验收

- 第一屏能否看出页面名称、当前上下文和主任务
- 是否没有并列多个同等级主操作
- 空态是否有现状说明，必要时是否给下一步动作
- 错误态是否说明发生了什么，并提供可执行的下一步
- 长列表是否考虑性能、返回定位和重复点击

## 表单验收

- 字段顺序是否符合用户理解顺序，而不是接口顺序
- 标签、值区、辅助说明、错误提示的位置是否稳定
- picker 的确认前是否不污染原值，取消后是否不误写回
- 提交失败后是否保留已有输入
- 键盘弹起后，当前字段、错误提示和主按钮是否仍可达

## 语音验收

- 项目内主录音模式是否统一
- 同一时刻是否只允许一个 active session
- 权限拒绝、录音中、转写中、失败态是否都有不同反馈
- 结果回填位置是否和触发入口一致
- 页面切换、组件卸载、取消录音后是否会清理脏状态

## 组件验收

- 是否真的有复用价值，而不是一次性布局被提前抽象
- props 是否先表达业务语义，而不是堆视觉开关
- `disabled`、`loading`、`error` 等关键状态是否完整
- 同类组件是否复用同一套 token、高度、圆角和间距

## 文案验收

- 是否说人话，而不是技术腔、营销腔或官话
- 是否先说结果，再说下一步
- 按钮是否使用明确动作词
- 失败提示是否给出用户可执行的后续动作

## 工程验收

- 重复点击、重复请求、重复录音是否被拦住
- 异步完成后回填前，是否校验上下文仍然有效
- 页面切换、弹层关闭、组件卸载后是否清理状态
- 需要吸底栏、弹层、滚动容器并存时，是否验证过真机行为

## 评审输出建议

做评审时，优先按下面顺序输出问题：

1. 行为回归或用户任务会失败的问题
2. 状态缺失或边界场景不稳定的问题
3. 交互和文案不一致的问题
4. 抽象过度或样式体系失控的问题

## 自检

- 本文件里的验收项是否能映射回已有专题文档
- 是否尽量使用可验证的条件，而不是纯抽象判断
- 是否覆盖了页面、表单、语音、组件、文案、工程这 6 类高频问题
