# 页面结构与流程

这个文件定义小程序页面的结构组织、信息层级和常见页面模式。

## 交互原则

- 操作路径短
- 主操作清楚
- 状态反馈及时
- 用户少思考一步就少思考一步

## 页面状态

关键页面至少考虑这些状态：

- loading
- success
- empty
- error
- retry

如果只实现成功态，说明页面还没设计完整。

## 页面壳

- 顶部区域承担标题、返回、身份摘要或当前上下文
- 主内容区承担信息展示和主要任务
- 底部操作区只放关键动作，不堆次要入口
- 吸底操作区必须考虑安全区和键盘遮挡
- 关键动作在对应场景下应保持可见，或能快速回到可见状态

## 列表页

- 列表页通常要提供搜索、筛选、排序中的至少一种能力
- 单行优先展示最关键字段
- 行内不要堆太多按钮
- 空列表必须有空态
- 长列表要考虑性能和返回定位体验

## 表单页的页面级规则

- 字段按业务理解顺序排，不按接口顺序排
- 主提交操作要稳定可见
- 键盘弹起不能挡住当前输入区和主按钮
- 页面级说明只说规则，不重复解释每个字段

字段级规则和反馈规则统一放在 `form-and-feedback.md`。

## 详情页

- 第一屏先给结论、状态和关键摘要
- 相关信息按组展示
- 操作按优先级排序
- 不要把详情页做成字段墙

## 我的页和设置页

- 常用入口放前
- 设置项按分组展示
- 风险操作单独放后面
- 身份摘要和设置列表分区清楚

## 弹层与确认

- 能在当前页完成的，不要强制跳转
- 能轻量确认的，不要做重弹层
- 删除、解绑、退出、覆盖关键数据等行为需要二次确认
- 不要把所有操作都做成确认框

## 交互禁区

- 把 hover 作为核心依赖
- 用复杂手势替代普通点击
- 让用户在多个弹层里来回跳
- 一屏里放多个同等级主操作

## 页面级自检

- 用户第一眼能否理解这个页面
- 主任务是否足够突出
- 失败时是否有清楚反馈
- 重复点击和弱网时是否仍然稳定
- 键盘弹起、安全区变化和长列表滚动后，主任务是否仍可完成
