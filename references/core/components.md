# 组件设计规范

这个文件定义通用小程序项目里的组件分层、边界和接口设计。

## 组件分层

### 基础组件

适合放这一层的：
- 按钮
- 输入框
- 文本域
- 标签
- 图标包装
- 分割线
- 加载占位

特点：
- 少业务语义
- 状态完整
- 为上层组件提供稳定基础

### 业务通用组件

适合放这一层的：
- 搜索栏
- 筛选栏
- 列表项
- 空态
- 确认弹层
- 吸底操作栏
- 表单字段组

特点：
- 有业务场景
- 能跨页面复用
- 接口优先围绕业务意图设计

### 场景组件

适合放这一层的：
- 日期范围选择器
- 地址选择器
- 上传区
- 语音录入区
- 审批面板
- 快速录入面板

特点：
- 上下文更强
- 只有在模式稳定且确认会复用时才抽

## 什么时候抽组件

满足以下条件中的至少两个再考虑抽：
- 同一模式出现两次以上
- 后续明显还会继续出现
- 页面里因为重复结构已经难维护
- 组件职责可以用一句话讲清楚

以下情况先不要抽：
- 一次性布局
- 视觉像，但交互和语义差很多
- 需要很多布尔开关才能适配

## 命名规则

- 按用途命名，不按长相命名

推荐：
- `SearchBar`
- `FilterPanel`
- `RecordListItem`
- `BottomActionBar`
- `EmptyState`

不推荐：
- `WhiteCard`
- `RoundBox`
- `BigBlock`

## props 规则

- 先暴露业务语义，再暴露视觉微调
- 用少量清楚的 props
- 默认值要覆盖大多数场景

推荐：
- `title`
- `description`
- `status`
- `disabled`
- `loading`
- `items`
- `value`

## 事件规则

- 事件名表达动作或结果

推荐：
- `confirm`
- `cancel`
- `change`
- `submit`
- `retry`
- `select`

## 状态完整性

可复用组件默认判断这些状态是否需要存在：
- 默认态
- 激活态
- 禁用态
- 加载态
- 错误态
- 空态

## 重点组件

### 按钮

- 同一页面通常只保留一个主操作视觉重心
- `loading` 时要防重复点击
- `disabled` 要明显
- 危险按钮要拉开语义差异

### 表单项

- 标签要清楚
- 错误提示贴近字段
- 必填信息明确
- 能选择就不强迫手输

### 列表项

- 第一眼先看到核心信息
- 点击区按触摸优先设计
- 行内按钮少而清楚
- 整行可点时避免局部点击冲突

### 空态

- 至少包含状态图形或图标
- 至少有一句现状说明
- 必要时给下一步动作

### 吸底操作栏

- 必须考虑安全区
- 必须考虑键盘弹起
- 不能遮挡核心内容

## 文案要求

- 组件内用户可见文案默认中文
- 不直接展示技术异常原文
