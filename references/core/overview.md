# 通用基线

这个文件定义小程序项目里跨页面、跨场景都应尽量保持稳定的基础规则。

## 语言和表达

- 默认使用中文输出、中文注释和中文交互文案
- 代码标识符、接口字段、数据库字段和第三方固定参数保留英文
- 用户可见文案优先说人话，不直接暴露底层技术信息

## 设计总原则

- 移动端优先
- 小程序优先
- 单手操作友好
- 快速扫读
- 低学习成本
- 稳定优先于花哨

## 落地优先级

1. 先继承项目现有规范
2. 再补齐缺失的页面和交互规则
3. 再统一组件和 token
4. 最后做局部视觉优化

如果项目已有成熟体系，本 skill 用来补缺失项，不用来推翻现有体系。

## 页面总原则

- 页面结构要稳定
- 第一屏要让用户知道自己能做什么
- 主操作要容易发现
- 顶部、主体、底部操作区职责要清楚
- 同一页面不要并列多个同等级主操作

## 反馈总原则

- 状态反馈及时
- 文案简短直接
- 先告诉用户发生了什么
- 必要时告诉用户下一步做什么
- 不把底层技术信息直接展示给用户

## 样式总原则

- 项目已有品牌规范时优先继承
- 项目没有规范时再建立少量稳定 token
- 颜色、圆角、间距、字号先语义化，再组件化
- 同一项目里不要并存多套竞争关系很强的样式体系

## 推荐 token 分层

推荐分为 4 层：
1. 基础值
2. 语义值
3. 组件值
4. 页面值

如果项目规模还小，最低也要先做到“基础值 + 语义值 + 少量关键组件值”。

不要一开始就把颜色和尺寸散写进页面组件。

## 新项目进入检查

进入一个新项目时，先看清这些东西：

- 有没有颜色、圆角、间距 token
- 有没有页面壳和导航模式
- 有没有统一的空态、加载态、错误态
- 有没有组件库或半成型组件层
- 有没有品牌文案口吻
- 有没有权限、提交态、失败态的默认处理方式

如果这些都没有，再用本 skill 的规则作为默认基线。

## 自检

- 是否先检查过项目现有规范，而不是直接套用本 skill
- 是否已经明确页面、反馈、样式和文案的统一口径
- token 分层是否至少达到“基础值 + 语义值 + 少量关键组件值”
- 当前项目是否已经有一套能复用的最小验收口径
